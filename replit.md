# HolaHola - Interactive Language Tutor

## Overview
HolaHola is an AI-powered language learning application providing interactive conversation practice, vocabulary building, and grammar exercises across ten languages, adhering to ACTFL standards. It offers personalized chat, flashcards, and adaptive grammar modules, all orchestrated by a unified AI named Daniela. The project's vision is to be a leading AI-driven educational platform, leveraging advanced technology and pedagogical innovation to capture significant market potential in personalized language acquisition.

## User Preferences
Preferred communication style: Simple, everyday language.
**DATABASE CONNECTION RULE (CRITICAL):** NEVER use `DATABASE_URL` or `process.env.DATABASE_URL` anywhere in the codebase. ALWAYS use `NEON_SHARED_DATABASE_URL` (`process.env.NEON_SHARED_DATABASE_URL`) for all database connections. This applies to all current code, bug fixes, and future development.
**SINGLE DATABASE (CRITICAL):** We use ONE shared Neon PostgreSQL database for BOTH development and production. There is no separate dev/prod database - all environments connect to the same database. This means database queries in "development" and "production" hit identical data.
Terminology standard: Use "Syllabus" in all user-facing text (database tables remain "curriculum*" for safety).
Batch doc updates: When user says "add to the batch" or "batch doc updates", add items to `docs/batch-doc-updates.md` for consolidated documentation updates later. **After completing any new feature**, add documentation to the batch doc covering: what was built, how it works, key files modified, and user-facing instructions.
Daniela development: Track personality/voice development in `docs/daniela-development-journal.md` using Honesty Mode → Founder Mode iteration cycle.
Neural network work: **REQUIRED READING** - `docs/neural-network-architecture.md` before any neural network changes. Prompts for context ONLY; neural network for procedures/capabilities/knowledge.
**TTS LANGUAGE SWITCHING (AUTO-DETECT):** Cartesia Sonic-3 always receives `autoDetectLanguage: true` — no explicit language code is sent. Sonic-3 handles mixed-language text natively with the same voice identity. Bold markers (`**word**`) from Gemini are still the source of truth for subtitle/karaoke highlighting via `segmentByLanguage()` in `language-segmenter.ts`, but segmentation is NOT used for TTS language selection anymore. The `detectTextLanguageForTTS()` fallback in `text-utils.ts` is no longer called in the main TTS path.
**GOOGLE TTS SINGLE INPUT (PROSODY FIX):):** `streamSynthesizeWithGoogle()` sends the entire text as ONE input to the bidirectional streaming API — no secondary sentence splitting. This preserves natural prosody and prevents fragmentation at internal punctuation (abbreviations, ellipsis, etc.). The orchestrator already splits at sentence boundaries before calling this function.
**GOOGLE BATCH TTS (CROSS-SENTENCE PROSODY):** When Google is the TTS provider, the PTT path uses "batch mode": Gemini generates ALL sentences first (collected in `batchedSentences[]`), then the orchestrator combines them into ONE paragraph and sends a single `streamSentenceAudioProgressive()` call. This gives Chirp 3 HD full cross-sentence context for natural prosody, matching Vertex Lab quality. Trade-off: audio playback starts after Gemini finishes generating (no progressive per-sentence audio). The `isGoogleBatchMode` flag gates this behavior in `streaming-voice-orchestrator.ts`. Other TTS providers (Gemini, Cartesia, ElevenLabs) continue using per-sentence progressive streaming with the lookahead pipeline.
**TTS BOLD MARKER PRESERVATION (CRITICAL FIX):** In the function-call path, `cleanTextForDisplay()` strips `**markers**` before text reaches TTS. Fix: `streamSentenceAudioProgressive()` accepts `preExtractedBoldWords?: string[]` — callers extract bold words from RAW text (before cleaning) and pass them in. Additionally, `(session as any).accumulatedBoldWords` collects bold words from ALL function calls in a turn (e.g., voice_adjust + drill), ensuring words from non-spoken function calls are still available for TTS segmentation. Accumulated words are cleared after each TTS call to prevent cross-turn leakage.
**NATIVE FUNCTION CALLS ONLY (CRITICAL):** Daniela uses ONLY Gemini native function calls. Never document or expect Daniela to write whiteboard tags like `[PLAY]...[/PLAY]` directly. Correct pattern: Gemini generates `play_audio({ description: "text" })` → Orchestrator converts to whiteboard update → Frontend renders. The `tool_knowledge` syntax field must use `FUNCTION CALL: function_name({ args })` format.
**NEW FUNCTION CALL CHECKLIST (CRITICAL — SIMPLIFIED VIA REGISTRY):** When adding a new Gemini function for Daniela: (1) Add a single entry to `DANIELA_FUNCTION_REGISTRY` in `daniela-function-registry.ts` — this defines the declaration, legacy type mapping, AND continuation response builder in one place. Declarations and command maps derive automatically. (2) Add `case 'MY_FUNC':` handler in `handleNativeFunctionCall()` in `streaming-voice-orchestrator.ts`. (3) If metadata-only (no whiteboard content), add required `text` param so Daniela produces audio. In handler, extract text to `(session as any).functionCallText` for TTS fallback. (4) Optionally add docs in `procedural-memory-retrieval.ts`. Functions using `spoken_text` instead of `text`: subtitle, show_overlay (where `text` is already used for display content). The old pattern of separately maintaining `DANIELA_FUNCTION_DECLARATIONS` array + `FUNCTION_TO_COMMAND_MAP` dict + duplicated PTT/OpenMic switch blocks is eliminated.
**UNIFIED CLASSROOM WORKSPACE:** Daniela's classroom (`classroom-environment.ts`) is the single spatial metaphor for ALL persistent session context. Folded INTO the classroom (not separate floating blocks): (1) Student Progress Board — student intelligence/ACTFL level/struggles/strategies, (2) Rehearsal Stage Notes — beta tester instructions/role reversal coaching, (3) System Status — voice health indicator, (4) Room Status — incognito mode atmosphere. Separate dynamic context (per-query): passive memories, identity memories, hive/express lane, text chat, editor feedback.
**WREN SECURITY OFFICER:** Wren serves as automated cybersecurity officer via `wren-security-audit-service.ts` (heuristic scanners) and `wren-security-audit-worker.ts` (scheduled worker). Posts to dedicated "Wren Security Officer" Hive session with compact summary + Gemini-enriched analysis. Wren Intelligence auto-captures findings as insights shared with Daniela via neural sync. Secret evidence always redacted before AI analysis. On-demand: `triggerSecurityAudit()`.
**LYRA LEARNING EXPERIENCE ANALYST:** Lyra is the platform's automated learning experience analyst via `lyra-analytics-service.ts` (data extraction + LLM analysis) and `lyra-analytics-worker.ts` (scheduled worker). ## System Architecture
The frontend is built with React, TypeScript (Vite), Shadcn/ui (Radix UI), and Tailwind CSS, supporting mobile-first, responsive Material Design with light/dark modes and PWA. Routing is handled by Wouter, and state management by React Context with TanStack Query. The backend uses an Express.js (Node.js) server with TypeScript, providing a RESTful API and Drizzle ORM for PostgreSQL.

Core AI orchestration is managed by a Unified TutorOrchestrator Architecture, centralizing interactions through Daniela. This includes the Hive Collaboration System and Student Learning Service. Key data models are Users, Conversations, VocabularyWords, and UserProgress. The system features a "Neural Network for Pedagogical Strategies," AI-powered conversation tagging, a Syllabus-Aware Competency System, and centralized Role-Based Access Control (RBAC). It supports pre-built syllabi across 10 languages and a unified ACTFL assessment. A Voice Diagnostics System provides TTS observability.

The Editor Intelligence System offers cross-session memory for the Replit Agent. The Alden Session Startup Protocol loads curated insights and recent conversation summaries. The Unified Daniela Context Service ensures a consistent "One Daniela always" experience by integrating all context sources.

Additional components include an Observation Summarization System, Daniela Content Growth System, Voice Intelligence System, and a Hybrid Memory Architecture for "infinite memory." A Message Checkpointing System prevents user message loss, and the ACTION_TRIGGERS Command Parsing System processes Daniela's literal tags for backend commands.

The Fluency Wiring System connects ACTFL Can-Do statements to lessons. An AI Lesson Generation System automatically creates structured lesson drafts, processed by a Lesson Publishing Service. The Drill System supports multiple interactive drill types, and the Practice Explorer System enables self-directed drill practice. An Interactive Textbook serves as a visual quick-reference. The Gauntlet Runner Identity Stress Test System validates Daniela's voice identity.

The TTS provider strategy supports Google Cloud TTS (Chirp 3 HD) as the recommended primary production provider, with Cartesia (Sonic-3), ElevenLabs (Flash v2.5), and Gemini (2.5 Flash Live) as alternatives. TTS dispatch is centralized via `tts-provider-adapter.ts` using a unified `TTSStreamingProvider` interface. Batch mode (Google) vs progressive streaming (others) is determined by the `adapter.requiresBatchMode` property. WebSocket warm-up and an app-level heartbeat ensure connection reliability.

The Voice Context Pipeline (`voice-context-pipeline.ts`) centralizes shared context-building logic for PTT and OpenMic voice paths, including classroom environment, passive memory search, identity memories, student intelligence, and dynamic preamble assembly.

The interactive textbook is accessible via `/interactive-textbook`.
Subject tutors for Biology (`/biology`) and History (`/history-tutor`) are live. Biology tutors are Evelyn and Gene; History tutors are Clio and Marcus. Their personas are defined in `server/services/biology-persona.ts` and `server/services/history-persona.ts`. The WS handler routes to the correct voice and system prompt based on session type.

Reading modules are pre-generated, permanently cached textbook documents for each curriculum topic. They are generated via a four-stage pipeline: OpenStax content fetching, structured JSON generation by Claude, academic citation enrichment by Perplexity, and quantitative claim verification by Wolfram. Modules are stored in the `reading_modules` table.

Reading Module UI: A `ReadingModulePanel` component (`client/src/components/ReadingModulePanel.tsx`) appears as a 384px right-side panel on the Biology and History tutor pages. It has a topic input, loading skeleton (20–30s on first generation), and renders: overview, Wikipedia image gallery (up to 3 images, tap to fullscreen), key concepts, key terms, misconceptions, framing questions, tap-to-reveal recall check, and citations. A standalone Reading Library page (`/reading-library`) shows all generated modules per subject with a searchable list, new topic generator, and inline module display. Accessible from the Library icon button on each tutor page header or from the sidebar under "Other Subjects". API: `GET /api/reading-modules/:subject` (list) and `GET /api/reading-modules/:subject/:topic` (get/generate). Images are fetched from Wikipedia REST API (CC BY-SA 4.0) using page summary + media-list endpoints; stored as `images[]` in the content JSONB. Existing modules without images still render cleanly.
