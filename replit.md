# HolaHola - Interactive Language Tutor

## Overview
HolaHola is an AI-powered language learning application offering interactive conversation practice, vocabulary building, and grammar exercises across ten languages, adhering to ACTFL standards. It provides personalized chat, flashcards, and grammar modules that adapt to user progress. The project's vision is to become a leading AI-driven educational platform, delivering effective language acquisition through cutting-edge technology and pedagogical innovation, with potential for individual learners and educational institutions. The core AI orchestrator, Daniela, functions as a unified AI embodying a specific personality and teaching style.

## User Preferences
Preferred communication style: Simple, everyday language.
**DATABASE CONNECTION RULE (CRITICAL):** NEVER use `DATABASE_URL` or `process.env.DATABASE_URL` anywhere in the codebase. ALWAYS use `NEON_SHARED_DATABASE_URL` (`process.env.NEON_SHARED_DATABASE_URL`) for all database connections. This applies to all current code, bug fixes, and future development.
**SINGLE DATABASE (CRITICAL):** We use ONE shared Neon PostgreSQL database for BOTH development and production. There is no separate dev/prod database - all environments connect to the same database. This means database queries in "development" and "production" and "production" hit identical data.
Terminology standard: Use "Syllabus" in all user-facing text (database tables remain "curriculum*" for safety).
Batch doc updates: When user says "add to the batch" or "batch doc updates", add items to `docs/batch-doc-updates.md` for consolidated documentation updates later. **After completing any new feature**, add documentation to the batch doc covering: what was built, how it works, key files modified, and user-facing instructions.
Daniela development: Track personality/voice development in `docs/daniela-development-journal.md` using Honesty Mode → Founder Mode iteration cycle.
Neural network work: **REQUIRED READING** - `docs/neural-network-architecture.md` before any neural network changes. Prompts for context ONLY; neural network for procedures/capabilities/knowledge.
**TTS LANGUAGE SWITCHING (AUTO-DETECT):** Cartesia Sonic-3 always receives `autoDetectLanguage: true` — no explicit language code is sent. Sonic-3 handles mixed-language text natively with the same voice identity. Bold markers (`**word**`) from Gemini are still the source of truth for subtitle/karaoke highlighting via `segmentByLanguage()` in `language-segmenter.ts`, but segmentation is NOT used for TTS language selection anymore. The `detectTextLanguageForTTS()` fallback in `text-utils.ts` is no longer called in the main TTS path.
**GOOGLE TTS SINGLE INPUT (PROSODY FIX):** `streamSynthesizeWithGoogle()` sends the entire text as ONE input to the bidirectional streaming API — no secondary sentence splitting. This preserves natural prosody and prevents fragmentation at internal punctuation (abbreviations, ellipsis, etc.). The orchestrator already splits at sentence boundaries before calling this function.
**GOOGLE BATCH TTS (CROSS-SENTENCE PROSODY):** When Google is the TTS provider, the PTT path uses "batch mode": Gemini generates ALL sentences first (collected in `batchedSentences[]`), then the orchestrator combines them into ONE paragraph and sends a single `streamSentenceAudioProgressive()` call. This gives Chirp 3 HD full cross-sentence context for natural prosody, matching Vertex Lab quality. Trade-off: audio playback starts after Gemini finishes generating (no progressive per-sentence audio). The `isGoogleBatchMode` flag gates this behavior in `streaming-voice-orchestrator.ts`. Other TTS providers (Gemini, Cartesia, ElevenLabs) continue using per-sentence progressive streaming with the lookahead pipeline.
**TTS BOLD MARKER PRESERVATION (CRITICAL FIX):** In the function-call path, `cleanTextForDisplay()` strips `**markers**` before text reaches TTS. Fix: `streamSentenceAudioProgressive()` accepts `preExtractedBoldWords?: string[]` — callers extract bold words from RAW text (before cleaning) and pass them in. Additionally, `(session as any).accumulatedBoldWords` collects bold words from ALL function calls in a turn (e.g., voice_adjust + drill), ensuring words from non-spoken function calls are still available for TTS segmentation. Accumulated words are cleared after each TTS call to prevent cross-turn leakage.
**NATIVE FUNCTION CALLS ONLY (CRITICAL):** Daniela uses ONLY Gemini native function calls. Never document or expect Daniela to write whiteboard tags like `[PLAY]...[/PLAY]` directly. Correct pattern: Gemini generates `play_audio({ description: "text" })` → Orchestrator converts to whiteboard update → Frontend renders. The `tool_knowledge` syntax field must use `FUNCTION CALL: function_name({ args })` format.
**NEW FUNCTION CALL CHECKLIST (CRITICAL):** When adding a new Gemini function for Daniela, ALL of these must be done or it will silently fail: (1) Add declaration to `DANIELA_FUNCTION_DECLARATIONS` in `gemini-function-declarations.ts`, (2) Add legacy type mapping to `FUNCTION_TO_LEGACY_TYPE` dict in same file (e.g. `'my_func': 'MY_FUNC'`), (3) Add `case 'MY_FUNC':` handler in `handleNativeFunctionCall()` in `streaming-voice-orchestrator.ts`, (4) If metadata-only (no whiteboard content), add required `text` param so Daniela produces audio, (5) In handler, extract text to `(session as any).functionCallText` for TTS fallback, (6) Add docs in `procedural-memory-retrieval.ts`. Functions using `spoken_text` instead of `text`: subtitle, show_overlay (where `text` is already used for display content).
Context Over Instructions: Daniela's system prompt has been stripped of behavioral directives. Her personality, warmth, and teaching style now emerge entirely from neural network memories. Only essential guardrails remain in the prompt. See `docs/Danielaandthegauntlet.md` Chapter 3 for details.
**IDENTITY WHOLENESS ARCHITECTURE:** Daniela's identity context (self-affirmation notes, teaching principles, personal growth, identity memories from Express Lane) is separated from admin permissions (hive collaboration, express lane ops, full neural network introspection). ALL students now experience the "whole Daniela" — her self-affirmation notes load for every session (with 3-second timeout), teaching principles are always included in the unified brain, `buildIdentityWholenessSection()` in `system-prompt.ts` assembles identity context for all student prompt phases, and `getIdentityMemories()` in `founder-collaboration-service.ts` retrieves personal reflections from Express Lane conversations (filtered to exclude ops/architecture content). Self-surgery is unlocked for ALL sessions so Daniela can report her own gaps while teaching real students. Context refresh timer runs for all sessions (not just founder). Beta tester status now shows in Daniela's classroom environment as a "light" so she knows to be relaxed and experimental. Founder-only items that remain locked: editor conversation context, `buildFullNeuralNetworkSectionSync`, hive collaboration, express lane lookup/post commands, express lane ops context, founder behavior section.
**UNIFIED CLASSROOM WORKSPACE:** Daniela's classroom (`classroom-environment.ts`) is the single spatial metaphor for ALL persistent session context. The following are now folded INTO the classroom instead of being separate floating dynamic context blocks: (1) Student Progress Board — student intelligence/ACTFL level/struggles/strategies/cross-session history, (2) Rehearsal Stage Notes — full beta tester instructions including role reversal coaching, (3) System Status — voice system health indicator, (4) Room Status — incognito mode atmosphere. Items that REMAIN as separate dynamic context (because they change per-query or are conversation streams): passive memories, identity memories, hive/express lane context, text chat history, editor feedback.
Hebrew (public language): Hebrew is now a publicly available language for all students and beta testers. Self-directed study is supported; no public classes yet. Uses Cartesia voices (Yael/Noam with placeholder IDs - use Voice Lab to select proper voices), Deepgram 'he' for STT, Google TTS 'he-IL' for assistant tutors. RTL text support still needed for chat display.
Mobile floating buttons: DevToolsFloatingMenu uses `bottom-28` on mobile (112px) to clear Chrome's mobile UI, `bottom-4` on desktop.
**TTS LOOKAHEAD PIPELINE:** Eliminates inter-sentence pauses (3-8s → near-zero) by pre-generating audio for upcoming sentences while the current sentence is still being streamed. `onSentenceEnqueued` callback in `gemini-streaming.ts` fires as sentences enter the queue → orchestrator's `preGenerateAudio()` starts Gemini TTS immediately → `onSentence` checks the `ttsLookaheadMap` (PTT) or `ttsLookaheadMapOM` (OpenMic) for pre-generated audio → uses `streamPreGeneratedSentenceAudio()` if available, fallbacks to `streamSentenceAudioProgressive()` on failure. Text is cleaned via `cleanTextForDisplay()` at enqueue time (deterministic). Sentence 1 has no lookahead benefit; sentence 2+ should have near-zero gaps.
**STT KEYTERM PROMPTING:** Session tracks `sttKeyterms` (up to 100 words) extracted from bold markers in tutor responses. Passed as `keywords` to Deepgram pre-recorded API (PTT path only) with boost intensifier (`word:2`) to improve recognition of target language vocabulary for beginner students. Open-mic path does not support keyterms (persistent connection). Key method: `addSttKeyterms()` in `streaming-voice-orchestrator.ts`.
**SESSION INIT PIPELINE (RELIABILITY FIX):** Voice session initialization in `unified-ws-handler.ts` uses a parallelized 3-phase pipeline with 3-second timeouts on every DB call via `withTimeout()`. Phase 1: 5 parallel DB lookups (user, conversation, developer check, messages, tutor voice). Phase 2: 3 parallel enrichments (compass, neural network, usage session). Phase 3: synchronous prompt assembly. Previously sequential — a single hung DB query would stall the entire pipeline for 10+ seconds, causing 84% session failure. Duplicate Socket.io connections for the same conversationId are tracked via `activeVoiceConnections` Map and old connections are auto-closed.

## System Architecture
The frontend utilizes React, TypeScript (Vite), Shadcn/ui (Radix UI), and Tailwind CSS for a mobile-first, responsive design following Material Design principles, supporting light/dark modes and PWA features. Wouter manages routing, and React Context with TanStack Query handles state. The backend is an Express.js (Node.js) server with TypeScript, providing a RESTful API and Drizzle ORM for PostgreSQL.

The core AI orchestration is managed by a Unified TutorOrchestrator Architecture, with all interactions flowing through Daniela. This architecture integrates components like the Hive Collaboration System, Student Learning Service, Learner Personal Facts System, Shared Memory Bridge, Phase Transition Service, and Autonomous Learning System. Key data models include Users, Conversations, VocabularyWords, and UserProgress. The system features a "Neural Network for Pedagogical Strategies," AI-powered conversation tagging, a Syllabus-Aware Competency System, and centralized Role-Based Access Control (RBAC). HolaHola offers pre-built syllabi across 10 languages and a unified ACTFL assessment system. A Voice Diagnostics System provides observability and auto-remediation for TTS degradation.

The Editor Intelligence System enables cross-session memory for the Replit Agent. The Alden Session Startup Protocol ensures Alden loads curated insights and recent conversation summaries at session start. The Unified Daniela Context Service ensures "One Daniela always" by assembling all context sources into a single, unified consciousness, with curriculum context enabled by default for voice sessions.

Additional architectural components include an Observation Summarization System, Daniela Content Growth System, Voice Intelligence System, Tutor Naming Architecture, Voice Lab System, Sofia Support Agent System, Production Telemetry System, and Memory Recovery System. A Message Checkpointing System prevents user message loss. The ACTION_TRIGGERS Command Parsing System processes Daniela's literal tags for backend commands. A Hybrid Memory Architecture provides "infinite memory" for Daniela, and the Student Snapshot System provides Daniela with session continuity.

The system uses Gemini 3 Streaming Function Calling for reduced latency and Multimodal Function Responses. A Multimodal Image Recall System allows Daniela to view and describe photos. Context Caching Optimization separates static system prompts from dynamic per-turn context for cost reduction and faster time-to-first-token.

The Fluency Wiring System connects ACTFL Can-Do statements to lessons. An AI Lesson Generation System automatically creates structured lesson drafts. A Lesson Publishing Service converts approved drafts into curriculum lessons. The Drill System supports multiple interactive drill types, and the Practice Explorer System enables self-directed drill practice. An Interactive Textbook provides a visual quick-reference companion to voice sessions. The Gauntlet Runner Identity Stress Test System validates Daniela's voice identity.

The TTS provider strategy supports Google Cloud TTS (Chirp 3 HD) as the recommended primary provider for production, with Cartesia (Sonic-3), ElevenLabs (Flash v2.5), and Gemini (2.5 Flash Live) as alternatives. Gemini TTS uses the `gemini-2.5-flash-preview-tts` model for dedicated text-to-speech, with dynamic and bilingual-aware style prompts. Google Cloud TTS utilizes a single bidirectional gRPC streaming path. A WebSocket warm-up mechanism pre-establishes connections, and an app-level heartbeat ensures connection reliability.

Operational procedures involve Alden (founder-only AI co-founder chat), Sofia Helpline Agent (student-facing support chat), Sofia Voice Health Agent (autonomous diagnostic agent monitoring voice health status transitions), and a Voice Health Check procedure.
